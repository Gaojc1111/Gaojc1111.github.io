MySQL中order by是怎么工作的

Hbzhtd2024-09-142024-09-17

前言

最近面试被问到一个问题，带where和order by子句的mysql，底层执行过程是怎样的？

自己上网搜了一些博客，并且翻了一下极客时间的《MySQL45讲》，正好有一篇文章是关于这个问题的，于是用自己的理解再梳理一遍。

对于上面这个语句，我们知道where子句是工作在order by之前的。

首先对where子句进行分析可知，如果city字段没有建立索引，那么整个sql会走全表扫描，效率是非常低的。

用explain语句查看这个语句的执行情况。

可以看到，Extra字段中有Using filesort，表明这条sql使用到了排序操作。

sort_buffer

另外一个知识点：MySQl会给每个线程分配一块内存用于排序，称之为sort buffer。

当需要排序的数据占用的空间小于sort buffer，排序操作就会全部在内存中执行；而当占用的空间大于sort buffer时，就会使用磁盘空间来辅助排序。

全字段排序

那么，这条sql语句的排序过程如下：

  1. 初始化sort_buffer，确定放入name、age、city三个字段；
  2. 从city索引找到第一个满足city=’guangzhou’ 条件的主键id；
  3. 根据主键id，在主键索引上找到完整的记录，取出name、city、age三个字段的值存入sort_buffer中；
  4. 再从city索引中取出下一个记录的主键id，如此往复。
  5. 当所有满足city = ‘guangzhou’ 当记录的数据被存放到sort_buffer时，按照字段name进行快速排序；
  6. 按照排序结果取前1000行返回给客户端，结束。

暂且把上述排序过程称之为全字段排序。

这里有一个问题，那就是’按照字段name进行快速排序’ 这个操作，可能在内存中就能完成，也可能需要使用到磁盘进行辅助排序，这就取决于sort_buffer_size 和 需要排序的数据的大小问题。

如果sort_buffer_size更大，就不需要使用磁盘；如果sort_buffer_size 更小，就需要借助磁盘辅助排序。

我们可以使用下面的方法，来查看该sql是否使用了磁盘作为辅助排序。

num_initial_chunks_spilled_to_disk表示的是，排序过程中使用的临时文件数。你一定奇怪，为什么需要 5个文件？内存放不下时，就需要使用外部排序，外部排序一般使用归并排序算法。

可以这么简单理解，MySQL 将需要排序的数据分成 5 份，每一份单独排序后存在这些临时文件中。然后把这 5 个有序文件再合并成一个有序的大文件。

如果 sort_buffer_size 超过了需要排序的数据量的大小，num_initial_chunks_spilled_to_disk 就是 0，表示排序可以直接在内存中完成。

否则就需要放在临时文件中排序。sort_buffer_size 越小，需要分成的份数越多，num_initial_chunks_spilled_to_disk 的值就越大。

rowid 排序

在上面这个算法过程里面，只对原表的数据读了一遍，剩下的操作都是在 sort_buffer 和临时文件中执行的。

但这个算法有一个问题，当查询返回的字段很多的话，那么 sort_buffer 里面要放的字段数太多，这样内存里能够同时放下的行数很少，要分成很多个临时文件，排序的性能会很差。

所以如果单行很大，这个方法效率不够好。

接下来，我们修改一个参数，让 MySQL 采用另外一种算法。

max_length_for_sort_data，是 MySQL 中专门控制用于排序的行数据的长度的一个参数。它的意思是，如果单行的长度超过这个值，MySQL 就认为单行太大，要换一个算法。

city、name、age 这三个字段的定义总长度是 36，我把 max_length_for_sort_data 设置为 16，我们再来看看计算过程有什么改变。

新的算法放入 sort_buffer 的字段，只有要排序的列（即 name 字段）和主键 id。

但这时，排序的结果就因为少了 city 和 age 字段的值，不能直接返回了，整个执行流程就变成如下所示的样子：

  1. 初始化 sort_buffer，确定放入两个字段，即 name 和 id；
  2. 从索引 city 找到第一个满足 city=’guangzhou’ 条件的主键 id；
  3. 到主键 id 索引取出整行，取 name、id 这两个字段，存入 sort_buffer 中；
  4. 从索引 city 取下一个记录的主键 id；
  5. 重复步骤 3、4 直到不满足 city=’杭州’条件为止；
  6. 对 sort_buffer 中的数据按照字段 name 进行排序；
  7. 遍历排序结果，取前 1000 行，并按照 id 的值回到原表中取出 city、name 和 age 三个字段返回给客户端。

这个执行流程，我把它称为 rowid 排序。

对比全字段排序，rowid排序在最后结尾部分，会多进行一次回表操作。

如果 MySQL 实在是担心排序内存太小，会影响排序效率，才会采用 rowid 排序算法，这样排序过程中一次可以排序更多行，但是需要再回到原表去取数据。

如果 MySQL 认为内存足够大，会优先选择全字段排序，把需要的字段都放到 sort_buffer 中，这样排序后就会直接从内存里面返回查询结果了，不用再回到原表去取数据。

因此，排序其实是一个成本比较高的操作。那么，难道所有的order by都要进行一次排序吗？非也，如果我们对city 和 name 字段建立一个联合索引，那么我们存入sort_buffer的数据就自然而然是按照name排好序的，此时就不用再做排序操作了。

这样整个查询过程的流程就变成了：

  1. 从索引 (city,name) 找到第一个满足 city=’杭州’条件的主键 id；
  2. 到主键 id 索引取出整行，取 name、city、age 三个字段的值，作为结果集的一部分直接返回；
  3. 从索引 (city,name) 取下一个记录主键 id；
  4. 重复步骤 2、3，直到查到第 1000 条记录，或者是不满足 city=’杭州’条件时循环结束。

看到这里就可以知道，整个流程是不需要进行排序操作的。再仔细思考，这个流程其实还可以进一步简化。

那就是使用索引覆盖。

如果我们对city，name，age三个字段作为联合索引，那么就不需要进行回表操作就可以直接拿到age的值，成本进一步降低。

这样整个查询语句的执行流程就变成了：

  1. 从索引 (city,name,age) 找到第一个满足 city=’杭州’条件的记录，取出其中的 city、name 和 age 这三个字段的值，作为结果集的一部分直接返回；
  2. 从索引 (city,name,age) 取下一个记录，同样取出这三个字段的值，作为结果集的一部分直接返回；
  3. 重复执行步骤 2，直到查到第 1000 条记录，或者是不满足 city=’杭州’条件时循环结束。

但是是否建立联合索引，这是一个需要权衡的决定，毕竟索引是需要维护代价的。
