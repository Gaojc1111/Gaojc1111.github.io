Go 八股题库

发表于 2023-09-25

  1. 相比较于其他语言, Go 有什么优势或者特点？

    • 语法简单易上手，代码风格统一

    • 通过goroutine和channel原生支持高并发。channel基于csp并发模型设计，即通过通道共享内存。

      通过GMP调度原理实现强大并发能力，降低了内核态和用户态的切换成本。

    • Go是跨平台编译的，编译出来的是二进制可执行文件，在不同系统上部署可直接运行

  2. Golang 中 make 和 new 的区别？（基本必问）

    • make只能用于初始化创建map，slice，chan类型的数据，返回的是Type
    • new可以用于创建任意类型的对象，返回一个指向该类型新地址的指针*Type
    • make 分配会初始化， new分配空间内存清零

  4. for range 的时候它的地址会发生变化么？

    地址不会发生变化，遍历时会对数组进行拷贝，然后对这个拷贝数组进行取值，再将值赋值给index和value，index和value被不断更新，但地址不变

    • for range 循环遍历 slice 有什么问题？
      1. 对slice用for range遍历取不到每个元素的地址
      2. 对slice用for range遍历，遍历过程中追加元素不会遍历到
      3. 对大的slice遍历会有性能问题，因为在遍历之前是对原数组做了一次拷贝

  5. go defer，多个 defer 的顺序，defer 在什么时机会修改返回值？（defer 和 return）

    1. 多个defer的执行顺序是栈（先进后出）

    2. defer and reurn ：

      当执行return 语句时，会先执行所有defer后执行拷贝

    uint是无符号整型

  8. Go中两个Nil可能不相等吗？

    可能不相等

    比如，接口interface{}和Nil

    两个接口进行比较时，先比较T（类型），在比较V（值）。

    接口和非接口比较时，会先将非接口转换成接口，在进行比较。

    上述中nil被转换成了(T=nil, V=nil)的接口，和(T=*int, V=nil)的i不一样，此时，接口类型的变量为nil和非接口类型的变量也为nil的时候，两者并不相等。

    2. uintptr并不是一个指针，它只是一个uint，可以进行指针运算，然后可以转换成unsafe.Point，再转换成普通指针，以此实现指针运算。

Context 相关：

  1. context 结构是什么样的？

    1. context是一个接口，其内部有四个方法，go语言为其提供了四个基本实现

    2. 四个基本实现：emptyCtx，cancelCtx，timerCtx，valueCtx

      1. emptyCtx：emptyCtx不具备任何功能，全部返回nil

        一般调用context.Background()和context.TODO()返回一个*emptyCtx

      2. cancelCtx：cancelCtx同时实现了Context和canceler接口

        一般调用context.WithCancel返回一个*cancelCtx和cancel()

      3. timerCtx：timerCtx内部封装了一个cancelCtx，同时含有一个deadline

        一般调用context.WithTimeout()返回timerCtx和cancel()，可以定时通知，也可以直接取消

        也可以调用context.WithDeadline()，在某个时间点通知，WithTimeout会转换成WithDeadline

      4. valueCtx：valueCtx内部封装了Context接口和一个键值对，实现数据传递

        一般调用context.WithValue()返回一个*valueCtx，可以继承父context的键值对

  2. context 使用场景和用途？（基本必问）

    1. 在goroutine之间传递信息，如传递请求的trace_id
    2. 通过cancel函数实现子协程的退出

Channel 相关：

  1. channel 是否线程安全？锁用在什么地方？

    1. channel是线程安全，其内部实现hchan中封装了一个锁
    2. 当对channel进行读写时，必须先获得锁，没有获得锁的goroutine进入阻塞

  2. go channel 的底层实现原理 （数据结构）

    1. 首先底层是一个hchan，包含了指向循环队列buf数组的指针，发送和接收元素的索引recvx和sendx
    2. 当goroutine对channel进行读写阻塞时，有读写两个队列， recvq，sendq
    3. 最后包含一个锁mutex，用于保护并发安全

  3. nil、关闭的 channel、有数据的 channel，再进行读、写、关闭会怎么样？（各类变种题型）

    操作\状态 未初始化（nil） 关闭（closed） 正常（normal）
    关闭 panic panic 关闭
    发送 永远阻塞，死锁 panic 成功发送或阻塞
    接收 永远阻塞，死锁 缓冲区内数据或零值 成功接收或阻塞

  4. 向 channel 发送数据和从 channel 读数据的流程是什么样的？

    1. 发送：
      1. 如果等待读取队列不为空，直接发送给队列第一个goroutine
      2. 如果等待读取队列为空，且循环队列未满，发送给循环队列队尾
      3. 循环队列满了，发送阻塞，将该goroutine加入等待发送队列，等待唤醒
    2. 接收
      1. 如果等待写入队列不为空并且是无缓冲，直接拷贝第一个等待队列中的数据，唤醒该goroutine
      2. 有缓冲，循环队列已满，将循环队列中的第一个数据拷贝，并且把等待写入队列的第一个goroutine数据拷贝到队尾，唤醒该goroutine
      3. 如果等待写入队列为空，并且buf数组为空，阻塞，挂起该goroutine
      4. buf数组不为空，拷贝第一个数据

Map 相关：

  1. map 使用注意的点，并发安全？

    map不是并发安全的，对map的并发读写需要加锁。

    对map进行查找，赋值，删除，遍历会对写标志检测，如果写标志为置位（1），则panic。否则对写标志位置位，再进行之后操作。

  2. map 循环是有序的还是无序的？

    无序的。

    map的遍历会随机选择一个桶位和槽位，从该桶位和槽位开始遍历，其他桶也从该桶位和槽位开始，

    为什么这样做？

    答：因为map会进行动态扩容，扩容之后key的位置会发生变化，再遍历时顺序就不一样了，因此go语言强制随机遍历

  3. map 如何顺序读取？

    可以开一个slice数组用于存储key，并且按照key排序。

    通过顺序遍历slice获取key再从map去获取value

  4. map 中删除一个 key，它的内存会释放么？

    不会，只是对key的value进行置空，不会释放内存。只有置空该map时，才会回收内存

  5. 怎么处理对 map 进行并发访问？有没有其他方案？ 区别是什么？

    对map加读写锁RWLock或者使用sync.Map

    sync.Map性能优于map，其底层用了两个map，read和dirty，空间换时间。可以无锁访问read并且优先操作。如果read满足要求就不用加锁访问dirty，减少了锁竞争频率。只有在read不满足才去加锁访问dirty

    优点：适合读多写少的环境

    缺点：在写多时，会导致read map缓存失效，需要加锁，锁竞争频率提高，性能急剧下降

  6. nil map 和空 map 有何不同？

    1. nil map 是未初始化的map，对其的增删改查会触发panic或死锁
    2. 空map是初始化后的但没有数据的map，对其增删改查不会报错

  7. map 的数据结构是什么？是怎么实现扩容？

    1. 底层数据结构是一个哈希表。表现为一个指向hmap的指针，hmap中包含了桶数组指针和溢出桶指针，以及元素个数等字段。每个桶是一个bmap的数据结构，其包含8个tophash，8个键值对和一个指向下一个溢出桶的指针overflow。为了内存紧凑，先存8个key，再存8个value

    2. 扩容

      1. 扩容时机：

        1. 负载因子（元素个数/桶个数） > 6.5：双倍扩容

        2. 溢出桶数量过多：等量扩容

        • 桶总数 < 2^15, 溢出桶 >= 正常桶 过多

        • 桶总数 > 2^15, 溢出桶 >= 2^15 过多

      2. 扩容机制：

        1. 双倍扩容：

          新建一个buckets数组，容量为原来两倍

        2. 等量扩容：

          不会扩大容量，只是把一些零散的键值对变得更加紧凑，提高bucket利用率，节省内存

      3. 扩容方式：

        扩容过程不是一次性的，是渐进式的。每次对key进行插入修改删除时，会检查olebuckets是否为nil，不为nil每次会搬迁两个桶

  8. map 的 key 为什么得是可比较类型的？

    当对map进行增删改时，会先通过key计算出hash值，然后通过hash值的低B位找到桶号，再根据高8位找到槽位，此时需要判断是否发生了哈希冲突，要对槽位上的key和当前的key进行比较，因此key必须是可比较类型

sync.map相关

  1. read map什么时候会更新？

    1. delete 和 update当key存在于read map中时
    2. misses >= dirty长度时，dirty提升为read，read更新

  2. dirty map什么时候会更新

    1. 插入新的key
    2. delete时，如果read中没有，dirty中直接删除
    3. 当修改read中的key发现这个key在dirty中不存在时，同步dirty
    4. 提升dirty时，更新完read后将dirty置空
    5. 当dirty为nil，插入key会重塑dirty，将read中不为expunged的所有key拷贝

  3. read map 和 dirty map的删除逻辑又什么区别？

    1. read map中删除key只是标记删除，没有真正的删除。将其value的指针置为nil，只有当dirty提升为read时才会对标记删除的key真正删除。
    2. dirty map中的删除是直接操控map的删除函数，是直接删除。

  4. 那既然在删除read的时候没有删除这个key，而dirty覆盖的时候又只覆盖了read，那么假如dirty中也存在这个key，这个key是不是会被遗漏，没有删掉，导致内存泄漏？

    答：不会。将dirty提升为read时，dirty置为nil，下次重塑dirty时，被标记删除的key成了expunged，dirty不会包含这个key，再一次提升dirty时，read中就不包含这个key了。

  5. sync.Map中的read和dirty有什么关系？

    答：

    read可以看作dirty的一个快照。dirty不为空时，dirty包含map中所有有效key，dirty为空时，read包含所有有效key。

    当read中misses >= dirty长度，提升dirty，将dirty中的key value覆盖read，将dirty置为nil。dirty为nil时，插入新的key，将会重塑dirty，将read中非expunged的key value又copy到dirty。

  6. sync.Map中的值是否一定是有效的

    不一定。值是由entry中的p指针指向的，p有三个状态，nil，正常值，expunged

    当p为nil或expunged时，此时这个值是被标记为删除，不是有效的值。

  7. sync.Map 应用场景

    1. 读多写少，因为加入新key时，要加锁操作dirty，效率变低
    2. 写操作多，但是修改的key和读取的key特别不重合

GMP 相关：

  1. 什么是 GMP？（必问）调度过程是什么样的？（对流程熟悉，要求更高，问的较少）

    1. GMP是go语言协程调度模型，g代表goroutine，m代表内核线程，p代表processor，p中包含g的本地队列，g通过p绑定到m才能执行
    2. 协程在刚创建时，会优先加入本地队列，等待调度。当这个p队列满了后，会将p的本地队列的一半g和新创建的g打乱顺序放入全局队列。每个m都有一个特殊的g0负责g的调度切换。调度如下：m优先执行p的本地g队列，如果本地没有，会先从全局队列获取，为了提高效率和负载均衡，会从全局获得多个p而非一个。当全局队列没有g时，会从其他m的p上偷取g，通常为该p队列的一半，如果还是没获取到，则m处于自旋状态

  2. GMP能不能去掉P层？会怎么样？

    答：不能。

    1. p层有自己的本地队列，可以减少m对全局队列的依赖，去掉p层会增加锁竞争。
    2. 每个p的本地队列数量相对平衡，在gmp中也实现了work stealing，可以减少空转，提高资源利用率

  3. M 和G、 P 的数量问题？

    1. P的数量由启动时环境变量$GOMAXPRPCS或者由runtime的方法GOMAXPROCS()决定
    2. M的数量默认时10000，go语言本身限制，但是内核支持不了这么大的数量，runtime/debug中的SetMaxThreads()函数可以设置M的最大数量
    3. G的数量理论上来说没有限制，受限于内存，但过多会影响性能

  4. 进程、线程、协程有什么区别？

    1. 进程可以理解为一个动态的程序。是操作系统资源分配的基本单位，而线程是操作系统资源调度的基本单位。进程独占一个虚拟内存单位，而进程里的线程共享一个进程虚拟内存单位。线程的粒度比进程更小，一个进程可以有多个线程。
    2. 协程可以理解为用户态线程
      1. 协程大小为2kb，可以动态扩容。线程大小为2mb。
      2. 线程切换需要用户态到内核态的切换，而协程只在用户态完成。线程切换需要保存各种寄存器，而协程切换只需要保存rsp，rip，rbp三个寄存器，切换消耗更小
      3. 线程的调度由操作系统完成，协程的调度由运行时的调度器完成

  5. 抢占式调度是如何抢占的？

    1. go1.14之前是基于协作式的抢占模式。go会创建一个线程用于运行sysmon函数，sysmon不需要p，sysmon函数会监测运行时间超过10ms的goroutine。如果超过，会将该goroutine的内部参数preempt设置为true，当goroutine进行函数调用时，会检查该参数，如果为true，会将自己与m进行分离并加入到全局g队列中，完成抢占。
    2. go1.14之后是基于信号的抢占。sysmon向运行了10ms以上的goroutine的p发送信号（SIGURG）。go的信号处理程序会调用p上的叫做gsignal的goroutine来处理该信号，将其映射到m而不是g，并检查该信号，gsignal发现抢占信号，停止正在运行的g
    3. 基于信号的抢占式调度可以避免类似for死循环的没有发生函数调用的goroutine一直阻塞占用cpu资源

  3. 原子操作和锁的区别

    1. 原子操作由底层硬件直接支持，锁是基于原子操作+信号量完成。原子操作更有效率
    2. 原子操作是单个指令的互斥，锁是一种数据结构，是多个指令的互斥，对原子操作的范围进行了扩展
    3. 原子操作是无锁操作，乐观锁；一般的锁都是悲观锁。

  4. Mutex 是悲观锁还是乐观锁？悲观锁、乐观锁是什么？

    Mutex属于悲观锁。

    1. 悲观锁：悲观锁认为在对数据进行操作时，无时无刻存在别人会修改数据，所以在对数据进行操作时会先上锁，直到操作完成再解锁。
    2. 乐观锁：乐观锁认为别人不会对数据更改，所以不会上锁，但是会在执行操作时检查数据有没有被修改，如果发现被修改，则放弃操作该数据。

  5. 互斥锁mutex底层是怎么实现的？

    mutex底层通过原子操作+信号量实现。通过atomic包中的一些原子操作进行锁的上锁，通过信号量实现阻塞和唤醒。

  6. Mutex 有几种模式？

    两种模式，正常模式和饥饿模式。

    当goroutine超过1ms没有获取到锁，会将该锁的starving设置成1，变成饥饿模式。

    模式 描述 公平性
    正常 goroutine按照先进先出顺序进行锁的获取，被唤醒的和新请求的竞争，新请求的更容易获取锁 否
    饥饿 所有goroutine排队等待，新请求的goroutine不会进行获取，而是加入等待队列 是

  7. goroutine 的自旋占用资源如何解决

    自旋指当goroutine获取锁时，锁已经被其他goroutine获取，该goroutine进入自旋状态，循环等待直到获取到锁。

    goroutine自旋要满足一定条件：

    1. 自旋没超过4次
    2. 锁已被占用，并且锁不处于饥饿模式
    3. 多核处理器
    4. GOMAXPROCS > 1
    5. p上本地goroutine队列为空

    mutex会让当前goroutine空转cpu，空转完成后，调用cas方法去尝试性的占有锁，直到不满足自旋，最终加入到等待队列，结束自旋。

  8. 读写锁底层是怎么实现的

    1. 加锁
      1. 读锁：若锁处于空闲，直接获取。若写锁锁定，则goroutine休眠
      2. 写锁：要用到mutex和readerWait。首先获取mutex，获取成功后，若readerWait > 0，说明读锁锁定，写锁阻塞；否则加写锁成功
    2. 解锁
      1. 读锁：直接释放。若有goroutine等待加写锁，释放读锁后readerWait减1，减到0时唤醒阻塞的写锁。
      2. 写锁：修改readerCount为正，解除互斥，唤醒所有读goroutine，然后释放mutex

  9. Mutex 已经被一个 Goroutine 获取了, 其它等待中的 Goroutine 们只能一直等待。那么等这个锁释放后，等待中的 Goroutine 中哪一个会优先获取 Mutex 呢?

    1. 正常模式下。当一个goroutine获取到锁时，其他goroutine会进入自旋模式（为了持有cpu）或者进入阻塞等待唤醒。但是新请求的goroutine自旋是仍然拥有cpu的，所以会比刚唤醒的goroutine更能获得锁。
    2. 饥饿模式下，新请求的goroutine会被加入到等待队列中，锁会给队列的第一个goroutine。

  10. waitgroup 是怎样实现协程等待？

    waitgroup内部维护了一个计数器，当执行wg.Add()函数,计数器加一；执行wg.Done()，计数器减一。

    当计数器减到0时，runtime_Semrelease会唤起wg.wait()阻塞的goroutine

  11. sync.Once 的原理，是怎样保证代码段只执行1次？

    其内部维护了一个标识位，当它==0时表示还没执行过，此时修改标识位，然后执行对应函数。后续再执行时，会发现标识位不为0，所以只会执行一次。

GC 相关：

  1. go gc 是怎么实现的？（必问）

    gc策略是采用三色标记法。但是单纯的三色标记会带来STW，在1.8版本之后，go采用三色标记配合混合写屏障技术实现gc

    追问：

    1. go gc经历了哪几个版本

      a. 1.3版本：普通标记清除法。整个gc过程都要STW，效率极低

      b. 1.5版本：三色标记法，堆空间启动写屏障，栈空间不启动，全部扫描后，要重新扫描栈（STW)，效率一般

      c. 1.8版本：三色标记配合混合写屏障技术：栈空间不启动（根节点可达和新加入的对象标记成黑色），堆空间启用写屏障，整个过程无需STW，效率高

    2. 三色标记法过程是怎样的？有什么问题？

      1. 刚开始扫描时，所有对象标记为白色。
      2. 从根节点开始，把能遍历的对象标记为灰色，然后遍历所有的灰色，把灰色能遍历的白色标记为灰色，然后把自身标记黑色，依次递推下去，直到灰色队列变为空

      三色标记过程中，可能出现用户将黑色对象引用白色对象，此时该白色对象可能会被误删，因此需要进行STW。

    3. 针对三色标记的STW，如何解决的？

      引用了屏障技术，主要分为插入写屏障和删除写屏障，1.8之后采用二者结合的混合写屏障

    4. 什么是插入写屏障？用插入写屏障解决STW会有什么问题？

      插入写屏障主要是针对新创建的对象或者是新添加引用关系的对象，被插入的对象或被引用的对象会被标记为灰色。

      但是插入写屏障只能在堆上操作，因为栈上函数调用十分频繁，在栈上开启写屏障会造成严重的性能问题。因此栈依然需要STW

    5. 删除写屏障是如何工作的？

      删除写屏障主要是断开引用关系时，将被断开的被引用的对象标记为灰色

    6. 混合写屏障是如何工作的？

      在栈上：gc刚开始时，栈上所有可达对象标记为黑色，在栈上创建新对象，均标记为黑色。这样保证了gc结束无需再对栈进行STW重新扫描。

      在堆上：被删除的和新添加的都标记为灰色

  2. GC 中STW时机，各个阶段是如何解决的？

    写屏障的开启和关闭需要STW

  3. GC 的触发时机？

    1. 可以在代码中调用runtime.GC手动触发
    2. 也可以由系统被动触发，当超过两分钟没有gc或内存分配达到一定的阈值就会强制触发gc

  4. GC扫描的根节点有哪些

    1. 全局变量：程序在编译器就能确定的那些存在于程序整个生命周期的变量
    2. 执行栈上的对象或指针：每个goroutine都包含自己的执行栈，这些执行栈上的对象包含栈上的变量及指向堆内存区块的指针
    3. 寄存器中的变量：寄存器的值可能表示一个指针，参与计算的这些指针可能指向某些赋值器分配的堆内存区块

内存相关：

  1. 谈谈内存泄露，什么情况下内存会泄露？怎么定位排查内存泄漏问题？

    1. 什么是内存泄漏？

      内存泄漏是程序生命周期的一些资源没有及时被回收，一直占用内存，导致这部分内存不可用

    2. 内存泄漏的原因？

      发生内存泄漏主要是因为程序阻塞或者空转导致程序不能及时结束。

      常见的情况：

      1. channel忘记初始化，读写阻塞
      2. channel发送数量超过接收数量
      3. channel接收数量超过发送数量
      4. http request body未关闭，goroutine不退出
      5. 互斥锁未解锁
      6. sync.WaitGroup使用不当
      7. time.Ticker未调用stop方法

    3. 如何排查内存泄漏？

      1. 单个函数：调用runtime.NumGoroutine()打印执行前后Goroutine的数量，进行前后比较
      2. 生产/测试环境：使用PProf实时监测Goroutine的数量

  2. 知道 golang 的内存逃逸吗？什么情况下会发生内存逃逸？

    内存逃逸是编译器根据逃逸分析策略，将原本应该分配到栈上的对象分配到堆上的一个过程。

    1. 什么情况会发生内存逃逸

      逃逸分析策略：

      1. 函数外部没有引用，优先放入栈
      2. 函数外部存在引用，必定放入堆
      3. 栈上放不下，必定放入堆

      逃逸场景：

      1. 方法内返回局部变量的指针
      2. 向channel发送指针数据
      3. 在闭包中引用包外的值，编译器无法确定闭包何时结束
      4. 在slice或map中存储指针
      5. 切片（扩容后）长度太大
      6. 在interface类型上调用方法

    2. 内存逃逸有什么影响？

      大量内存逃逸会给gc带来压力

  3. 请简述 Go 是如何分配内存的？

    go语言对象的分配根据对象大小不同分配策略也不同：

    1. 对象 <= 16B时，从mcache上的微型分配器上分配
    2. 对象 >16B && 对象 <= 32K时，从P上的mcache分配，如果mcache不够，从mcentral获取，如果mcentral没有，向mheap申请，如果mheap么有，向操作系统申请
    3. 对象 > 32K时，从mheap申请

  4. Channel 分配在栈上还是堆上？哪些对象分配在堆上，哪些对象分配在栈上？

    1. channel分配在堆上，因为channel用于协程间的通信，作用域和生命周期不可能局限于某个函数内部，所以go直接将其分配到堆上
    2. 一般来说，大的对象分配在堆上，小的优先分配到栈上。如果满足内存逃逸策略，应该分配到栈上的也会分配到堆上。

  5. 介绍一下大对象小对象，什么情况下会导致GC压力大？

    1. go中 <= 32ke的对象为小对象，< 16B的是微小对象（而且是不含指针的对象），其他都是大对象
    2. 当大量的小对象逃逸到堆上，或者有很大的map或slice，GC压力会比较大
